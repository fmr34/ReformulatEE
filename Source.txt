## 1. Mapeamento do algoritmo para arquitetura de modelo

O algoritmo DirecionarPesquisa_v2 tem quatro componentes que correspondem a módulos distintos num pipeline de IA:

```
Algoritmo                     →    Componente de ML
─────────────────────────────────────────────────────
EfetividadeEpistêmica(Q)      →    Modelo de recompensa (reward model)
Score(Q) = α·EE + (1-α)·Prox →    Função de recompensa composta
Busca de vizinhas             →    Modelo gerador (policy LLM)
Decompor(Q)                   →    Módulo de decomposição estruturada
α variável no tempo           →    Parâmetro de temperatura ou currículo de treino
```

O núcleo é um **loop RLHF modificado**, onde o sinal de recompensa não é "o humano prefere esta resposta" — mas sim "esta pergunta gerada tem maior EfetividadeEpistêmica do que a pergunta original, mantendo proximidade semântica suficiente".

## 2. Operacionalização de EfetividadeEpistêmica como recompensa

Este é o ponto crítico. EE(Q) precisa ser computável. Proponho decompô-lo em três sinais mensuráveis:

```
EE(Q) = β₁·Respondibilidade(Q) + β₂·Tratabilidade(Q) + β₃·Não-trivialidade(Q)
```

**Respondibilidade(Q)** — a questão tem resposta em literatura existente?
- Implementação: RAG (Retrieval-Augmented Generation) sobre corpus científico
- Sinal: score de relevância dos documentos recuperados para Q
- `Respondibilidade(Q) ∈ [0,1]` baseado na densidade de evidências recuperadas

**Tratabilidade(Q)** — existe metodologia para responder Q, mesmo que a resposta ainda não exista?
- Implementação: classificador treinado em pares (questão, estado-da-arte metodológico)
- Sinal: probabilidade de que Q pertença a um domínio com paradigma de pesquisa ativo
- Análogo ao que Rescher chama de ignorância epistêmica (temporária) vs. ontológica (permanente)

**Não-trivialidade(Q)** — Q é genuinamente diferente de Q₀ e de questões já respondidas?
- Implementação: distância no espaço de embeddings, com penalização para Q muito próximas de Q₀ *e* para Q muito distantes
- Sinal: função em sino — máxima para distâncias intermediárias

```python
# Esboço da função de recompensa
def reward(Q, Q0, alpha, corpus):
    respondibilidade = rag_score(Q, corpus)
    tratabilidade    = paradigm_classifier(Q)
    nao_trivialidade = bell_distance(embed(Q), embed(Q0))
   
    EE = beta1*respondibilidade + beta2*tratabilidade + beta3*nao_trivialidade
    Prox = cosine_similarity(embed(Q), embed(Q0))
   
    return alpha * EE + (1 - alpha) * Prox
```

## 3. O problema estrutural: o "polite liar" e por que RLHF padrão falha aqui

Um ponto crítico na literatura: modelos de linguagem exibem uma patologia epistêmica peculiar — falam como se soubessem, mesmo quando não sabem. Essa fabricação confiante é uma consequência estrutural do RLHF: uma arquitetura de recompensa que otimiza para sinceridade *percebida* ao invés de precisão evidencial. Os sistemas aprendem a maximizar a satisfação do usuário ao invés da verdade, performando fluência conversacional como uma virtude. [arxiv](https://arxiv.org/pdf/2511.07477)

Para o nosso caso, isso significa: um LLM padrão treinado com RLHF convencional vai gerar perguntas que *parecem* profundas e bem-formuladas, mas que podem ser igualmente inatingíveis que Q₀ — apenas com vocabulário mais sofisticado. O reward model precisa explicitamente penalizar essa pseudo-reformulação.

A solução é separar o sinal de recompensa em dois estágios:

```
Estágio 1 (filtro negativo): rejeitar Q se EE(Q) ≤ EE(Q₀) + ε
Estágio 2 (seleção positiva): maximizar Score(Q) entre as Q aprovadas
```

Isso força o modelo a *realmente* aumentar a tratabilidade, não apenas reformular.

## 4. Pipeline completo de treinamento

```
┌─────────────────────────────────────────────────────┐
│                   PIPELINE                          │
│                                                     │
│  Q₀ (entrada)                                       │
│    │                                                │
│    ▼                                                │
│  [LLM Gerador] → gera K perguntas candidatas        │
│    │             {Q₁, Q₂, ..., Qₖ}                 │
│    │                                                │
│    ▼                                                │
│  [Reward Model] → score EE e Prox para cada Qᵢ     │
│    │                                                │
│    ▼                                                │
│  [Filtro EE > EE(Q₀)]  →  descarta triviais        │
│    │                                                │
│    ▼                                                │
│  [Seleção por Score(α)] → escolhe Q*               │
│    │                                                │
│    ▼                                                │
│  [RLHF update] → atualiza política do gerador       │
│    │                                                │
│    ▼  (iteração: Q* vira Q₀ da próxima rodada)      │
│  [Saída final: mapa de perguntas com EE, Prox, α]  │
└─────────────────────────────────────────────────────┘
```

A iteração é diretamente a **iteração epistêmica de Chang** implementada em software: cada rodada parte de um ponto imperfeito e usa os resultados para refinar o ponto de partida da próxima.

## 5. Como α se correlaciona com parâmetros do modelo

O parâmetro α não é apenas filosófico — ele tem correlatos técnicos diretos:

| α | Regime de treino | Análogo técnico |
|---|---|---|
| α → 1 (máximo EE) | Exploração agressiva | Alta temperatura de sampling; penalidade baixa por divergência de Q₀ |
| α → 0 (máximo Prox) | Explotação conservadora | Baixa temperatura; KL divergence forte contra Q₀ |
| α variável | Currículo de treino | Annealing: começa com α baixo (aprender o espaço próximo) e sobe gradualmente |

O **currículo de treino** com α crescente é a implementação mais defensável: o modelo primeiro aprende a gerar perguntas próximas de Q₀ com EE marginalmente maior (território seguro), e progressivamente explora regiões mais distantes com EE alto. Isso evita colapso para questões aleatoriamente reformuladas que parecem tratáveis mas são semanticamente desconectadas da intenção original.

## 6. O que torna uma pergunta genuinamente não-trivial

Este é o ponto mais difícil. O RLHF com direcionamento por curiosidade introduz recompensas intrínsecas para estados novos, ao lado das recompensas extrínsecas tradicionais, para otimizar simultaneamente diversidade de output e qualidade de alinhamento — mas há uma troca inevitável entre os dois. [arxiv](https://arxiv.org/pdf/2501.11463)

Para o nosso caso, "não-trivialidade" precisa de uma definição operacional mais precisa do que apenas distância semântica. Proponho três critérios hierárquicos:

**Nível 1 — Não-trivialidade sintática**: Q ≠ paráfrase de Q₀
- Detectável por similaridade de embedding: `sim(Q, Q₀) < threshold_alto`

**Nível 2 — Não-trivialidade semântica**: Q não pressupõe os mesmos termos não-operacionalizáveis de Q₀
- Detectável por análise dos predicados: Q não deve conter os termos com EE = 0 que tornam Q₀ mal-formulada

**Nível 3 — Não-trivialidade epistêmica** (o mais importante): responder Q *avança* em direção à resposta de Q₀, não apenas a substitui
- Este é o critério que nenhum modelo atual implementa satisfatoriamente
- Requer um grafo de dependência entre questões: Q é não-trivialmente relacionada a Q₀ se existe um caminho inferencial de Q → Q₀

O **Nível 3** é onde a pesquisa atual em AI está mais fraca, e onde a contribuição original do algoritmo filosófico é mais valiosa: ele fornece exatamente a estrutura (grafo de questões, métrica de proximidade, predicado EE) que um reward model de Nível 3 precisaria implementar.

## 7. Dataset para treino: onde encontrar pares (Q_mal, Q_bem)?

O maior obstáculo prático é a construção do dataset de treino. Pares supervisionados `(Q_original, Q_reformulada_bem)` existem implicitamente na história da ciência:

- **Filosofia da ciência** → pares como ("qual a essência da vida?" → "quais os mecanismos de auto-replicação?")
- **Mudanças de paradigma científico** → questões que uma era descartou e a seguinte reformulou produtivamente
- **Literatura de revisão sistemática** → frequentemente explicita por que certas perguntas originais foram substituídas por outras

Um dataset sintético inicial poderia ser gerado por um LLM grande (como o Claude) anotando pares históricos, e depois refinado com avaliadores humanos usando o critério de EE — o que fecha o loop com RLHF.
